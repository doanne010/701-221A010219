Giải thích logic tư duy cho từng bài tập

Câu 1 — Danh sách sản phẩm & Tìm kiếm

1. Xử lý input người dùng để tránh lỗi injection

Khi dữ liệu được chèn vào DOM qua innerHTML, trình duyệt sẽ hiểu các ký tự < và > như thẻ HTML thật. Nếu không xử lý, một chuỗi tìm kiếm độc hại có thể gây ra lỗi XSS (Cross-Site Scripting). Giải pháp là viết hàm escapeHTML() chuyển các ký tự nhạy cảm (& < > " ') thành HTML entity trước khi đưa vào innerHTML:

function escapeHTML(str) {
  const map = { "&":"&amp;", "<":"&lt;", ">":"&gt;", '"':"&quot;", "'":"&#39;" };
  return String(str).replace(/[&<>"']/g, ch => map[ch]);
}

Ngoài ra, vì chức năng highlight từ khóa cần tạo RegExp động từ input người dùng, các ký tự đặc biệt của regex (. * + ? ( )...) cũng được escape riêng bằng replace(/[.*+?^${}()|[\]\\]/g, "\\$&") để tránh lỗi cú pháp hoặc hành vi tìm kiếm sai khi người dùng gõ ký tự đặc biệt.

2. Tối ưu tìm kiếm

  • Không phân biệt hoa thường: cả từ khóa và tên sản phẩm đều được chuẩn hóa bằng .toLowerCase() trước khi so sánh bằng .includes().
  • Debounce: thay vì lọc mảng ngay mỗi lần gõ phím, dùng setTimeout/clearTimeout để trì hoãn 200ms — chỉ thực sự lọc khi người dùng ngừng gõ, giảm số lần tính toán và render lại DOM.
  • Thông báo lỗi: nếu mảng kết quả lọc rỗng, bật class CSS .show cho phần tử thông báo (mặc định ẩn bằng display:none) thay vì gán style trực tiếp, giúp tách biệt logic và giao diện.

Câu 2 — Form đăng ký người dùng

1. Xử lý sự kiện submit

Form dùng e.preventDefault() trong sự kiện submit để ngăn trình duyệt tải lại trang (hành vi mặc định của thẻ <form>), từ đó toàn quyền kiểm soát luồng xử lý bằng JavaScript. Việc validate được tách thành các hàm nhỏ, kiểm tra từng trường độc lập (họ tên, email, mật khẩu, checkbox điều khoản) và gom kết quả vào một biến valid chung — nếu bất kỳ trường nào sai, form sẽ không submit và hiển thị lỗi tương ứng ngay dưới từng input thay vì chỉ báo lỗi chung chung.

const EMAIL_REGEX = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;

function isStrongPassword(pw) {
  return pw.length >= 8 && /[A-Z]/.test(pw)
      && /[a-z]/.test(pw) && /[0-9]/.test(pw);
}

Quy tắc mật khẩu mạnh được kiểm tra bằng 4 điều kiện độc lập: độ dài tối thiểu 8 ký tự, có ít nhất 1 chữ hoa, 1 chữ thường và 1 chữ số — dùng các regex riêng biệt để dễ đọc và dễ thông báo lỗi cụ thể cho người dùng (thay vì 1 regex phức tạp khó bảo trì).

2. Bảo mật dữ liệu cục bộ (LocalStorage)

LocalStorage lưu dữ liệu dạng văn bản thuần, không mã hóa, và có thể bị đọc bởi bất kỳ đoạn JavaScript nào chạy trên cùng origin (kể cả mã độc nếu trang bị XSS) — do đó không bao giờ lưu mật khẩu dạng rõ (plain text). Giải pháp áp dụng trong bài là băm (hash) mật khẩu bằng thuật toán SHA-256 thông qua Web Crypto API trước khi lưu:

async function hashPassword(password) {
  const data = new TextEncoder().encode(password);
  const hashBuffer = await crypto.subtle.digest("SHA-256", data);
  return Array.from(new Uint8Array(hashBuffer))
    .map(b => b.toString(16).padStart(2, "0")).join("");
}

Lưu ý quan trọng: đây chỉ là giải pháp demo phía client để minh họa nguyên tắc "không lưu plain text". Trong ứng dụng thực tế, việc xác thực và lưu trữ mật khẩu phải luôn được thực hiện ở phía server (với thuật toán băm chuyên dụng như bcrypt/argon2 và salt riêng), vì bất kỳ logic nào chạy hoàn toàn trên trình duyệt đều có thể bị người dùng xem hoặc can thiệp mã nguồn. Ngoài ra, dữ liệu họ tên và email cũng được escape qua escapeHTML() trước khi lưu, để nếu sau này dữ liệu này được hiển thị lại ra trang khác, nó vẫn an toàn trước XSS.

Câu 3 — Đồng hồ đếm ngược

1. Xử lý interval để tránh memory leak

Lỗi phổ biến nhất khi dùng setInterval là gọi nó nhiều lần (ví dụ người dùng bấm nút "Bắt đầu" liên tục) mà không hủy interval cũ, dẫn đến nhiều interval chạy song song — đồng hồ sẽ đếm nhanh hơn bình thường và tốn tài nguyên không kiểm soát được, đây chính là một dạng memory leak. Giải pháp là luôn giữ một biến duy nhất tham chiếu tới interval đang chạy (intervalId), và gọi clearInterval() trước khi tạo interval mới:

let intervalId = null;

function startTimer() {
  clearTimer(); // luôn dọn interval cũ trước
  intervalId = setInterval(() => {
    remainingSeconds--;
    updateDisplay();
    if (remainingSeconds <= 0) clearTimer();
  }, 1000);
}

function clearTimer() {
  if (intervalId !== null) {
    clearInterval(intervalId);
    intervalId = null;
  }
}

Ngoài ra, interval còn được dọn dẹp ở 2 thời điểm khác: khi đồng hồ đếm về 0 (tự động dừng), và khi người dùng rời trang thông qua sự kiện beforeunload — đảm bảo không có interval nào tiếp tục chạy ngầm sau khi DOM liên quan đã bị hủy.

2. Tích hợp với Domain/Hosting khi deploy (đồng bộ thời gian server)

setInterval dựa vào đồng hồ hệ thống của trình duyệt người dùng, nên nếu máy người dùng chỉnh sai giờ, hoặc tab bị chuyển nền (background throttling) khiến interval bị trình duyệt làm chậm lại, thời gian đếm ngược hiển thị có thể bị lệch so với thực tế. Khi deploy lên hosting thật (ví dụ GitHub Pages) và cần độ chính xác cao hơn (như đếm ngược tới một mốc thời gian sự kiện chung cho nhiều người dùng), cách tối ưu là:

  • Không tính thời gian còn lại bằng cách trừ dần (remainingSeconds--) mỗi giây, mà tính lại từ một mốc thời gian kết thúc cố định (endTime) lấy từ server, ví dụ qua một API trả về thời gian hiện tại của server lúc tải trang.
  • Mỗi lần tick, tính remainingSeconds = Math.round((endTime - Date.now()) / 1000) thay vì trừ dần, để tự động bù trừ cho các lần tick bị trễ do tab nền hoặc máy chậm.
  • Định kỳ (ví dụ mỗi 30 giây - 1 phút) gọi lại API server để đồng bộ lại độ lệch giữa đồng hồ client và server, tránh tích lũy sai số theo thời gian.

Vì bài tập này không có backend riêng, đồng hồ hiện dùng thời gian cục bộ của trình duyệt (đủ chính xác cho mục đích demo cá nhân), nhưng kiến trúc hàm startTimer()/clearTimer() được tách riêng để dễ dàng thay thế phần tính giờ bằng logic đồng bộ server nếu cần mở rộng sau này.
